1. What we built

A modal wizard that lets a user import time-series weather data from a CSV / TXT / XML file into the Helios backend. The user picks a file, confirms how the file is shaped, maps date/time components, reviews which columns to import, and clicks Import.

End-to-end the wizard handles four moving parts:

  1. File picking — native dialog, read raw bytes
  2. Parsing — auto-detect delimiter / header skip; or XML record extraction
  3. Date-time mapping — Group 1 (Y/M/D/H/M columns) or Group 2 (Date + Time columns)
  4. Backend upload — single POST /addCol call with all values inline

Plus a synthetic check column injected on import (default '1') so downstream tools can include/exclude rows later.

3. Data lifecycle

Five distinct shapes, four boundaries:

StageFormatWhere it livesHow it's produced
1. Raw file bytes string state.weather.pickedFile.rawText (Redux) Saga calls window.api.openFile + readFile; main process reads from disk
2. Parsed table ParseResult JS object Wizard local useState parseFile() runs render-time when pickedFile changes
3. Wizard intermediate state multiple useState values Wizard local User input on Steps 2–4 (delimiter, mode, mapping, columnSelection)
4. ImportedDataset JS object, fully serializable state.weather.dataset (Redux, after success) handleImport() on Step 4 builds it from local state
5. Backend payload JSON body to POST /addCol Network request Saga reshapes ImportedDataset into per-column values: [{date, time, value}]

Concrete shapes

ParseResult (Stage 2)

{
  format: 'csv',                    // 'csv' | 'txt' | 'xml'
  delimiter: ',',                   // empty for XML
  headerLinesToSkip: 0,
  headers: ['year', 'month', 'day', 'hour', 'temp'],
  rows: [
    ['2026', '2', '26', '10', '22.5'],
    ['2026', '2', '27', '11', '23.7']
  ]
}

ImportedDataset (Stage 4)

{
  filename: 'sample.csv',
  columns: [
    { key: '__check__', label: 'check', index: -1 },
    { key: '5__temp',   label: 'temp',  index: 5  }
  ],
  records: [
    { dtIso: '2026-02-26T10:00:00.000Z', values: { __check__: '1', '5__temp': '22.5' } },
    { dtIso: '2026-02-27T11:00:00.000Z', values: { __check__: '1', '5__temp': '23.7' } }
  ]
}
Why the funny key shape 5__temp? CIMIS-style files can have duplicate header names (e.g. multiple qc columns). Keying values by ${index}__${header} makes them addressable independently. The label field carries the human-readable name for display and for the backend.

4. Major design decisions

4.1 — Where do the parser functions live?

LocationStatusProsCons
containers/Weather/parsers.ts (single file) Chosen Co-located with the only feature using it; single 500-line file is cohesive; matches existing container shape (no nested folders) Slightly longer file
utils/weatherImport/ (folder of 8 files) Rejected Separation by concern (parseDelimited.ts, parseXml.ts, …); easier to grep individually Would be the first nested module under utils/ — breaks the "flat single-file" pattern that api.ts, format.ts, sse.ts follow
utils/weatherImport.ts (single flat file) Rejected Matches utils convention Treats weather-specific logic as cross-cutting when it isn't (only the wizard uses it)

4.2 — Where does the wizard component live?

LocationStatusReasoning
components/ImportWizard/ Chosen Project convention: presentational components live under components/. The wizard is a component that the Weather container mounts. User explicitly directed this location.
Inside containers/Weather/ Rejected Existing containers don't have nested component subfolders. Mixing presentational components into containers breaks the established split.

4.3 — Action shape: interface or type?

FormStatusWhy
type ImportResetAction = { type: typeof IMPORT_RESET } Chosen Type aliases are structurally closed — TS treats them as assignable to indexed types. Required for Redux 5's useDispatch, which returns Dispatch<UnknownAction> where UnknownAction = { type: string; [k: string]: unknown }.
interface ImportResetAction { type: typeof IMPORT_RESET } Rejected Interfaces are open for declaration merging — TS conservatively rejects assigning them to indexed types. Caused real compile errors when we tried it: "Index signature for type 'string' is missing".
No explicit return type, infer from () => ({ type, payload }) Rejected HomePage uses this style; works fine. But Weather already had explicit interfaces; converting to type aliases kept the file consistent.

4.4 — State strategy per piece

State pieceWhere it livesWhy
stepIdx, mode, mapping, dateFormat, columnSelection Wizard local useState Pure UI state. No external consumer. Resets when modal unmounts via parent's conditional render ({showWizard && <ImportWizard />}). Putting it in Redux would mean 8+ action types for transient state.
parsed (the ParseResult) Wizard local useState Derived from pickedFile via render-time pattern (see 4.5). Survives delimiter / header-skip changes that re-parse from rawText.
pickedFile, fileLoading, fileError Redux state.weather Saga side effect produces them. Wizard reads as props. Cleared on close via IMPORT_RESET.
importing, importError, dataset Redux state.weather Result of the API call must survive modal close. Auto-close logic in container watches importing transition.

4.5 — Reactive parsing: useEffect or render-time pattern?

When pickedFile arrives from Redux, the wizard needs to parse it. Two options:

PatternStatusTrade-offs
Render-time setState (if (pickedFile !== lastSeen) { setLastSeen(pickedFile); setParsed(...); }) Chosen React-recommended pattern for "derive state from props." Avoids a setState-in-effect cascade. Lints clean under react-hooks/set-state-in-effect.
useEffect(() => { setParsed(parseFile(...)) }, [pickedFile]) Rejected The conventional pattern, but React 19+ ESLint rule flags it: it cascades an extra render. Render-time pattern is the official replacement.

4.6 — Backend upload: one call or two?

The backend specs two endpoints: /addCol for column registration, then /addRow for the data. We tried it and hit a backend bug.

ApproachStatusWhy
Single /addCol with all data inline in values: [{date, time, value}] Chosen Backend's /addCol with empty values writes columns to SQL but doesn't register them in PyHelios. So /addRow rejected every key as unknown. Putting data inline forces PyHelios registration — works in one call.
Two-step: /addCol empty → /addRow Backend bug The "by-the-book" two-step shape. Should be revisited if/when backend is fixed.
Save to disk via saveFile/writeFile Earlier interim Used during development before the API existed. Let us verify the CSV output by opening it.

4.7 — Auto-close on success

PatternStatusWhy
Render-time prev/curr comparison: if (prevImporting !== importing) { setPrevImporting(importing); if (prev && !curr && !err) setShowWizard(false); } Chosen Same render-time pattern as 4.5. Avoids the lint warning.
useEffect with useRef for prev value Rejected Cleaner-looking but tripped set-state-in-effect. HomePage uses this pattern but for some reason gets a pass — render-time was the safer port.
Dispatch a "close wizard" action from the saga directly Rejected Couples backend success to UI state. UI controls visibility; saga controls data — clean split preserved.

4.8 — Lazy-loading the wizard

ApproachStatusWhy
const ImportWizard = loadable(() => import('@renderer/components/ImportWizard')) Chosen The wizard pulls in 9 source files + parsers (~25 KB minified). Code-split keeps the Weather screen's initial chunk small. The bundle only fetches when the user first clicks "Import Weather Data".
Direct import Rejected Pulls the entire wizard into Weather's chunk on first render — wasted bytes if the user never clicks Import.

4.9 — Date format strictness

ComponentStrictnessReasoning
Date (Y/M/D in Group 1; Date column in Group 2) STRICT Year must be exactly 4 digits in source — "26" is ambiguous, rejected. Month 1–12, Day 1–31. Bad → row is invalid_date, dropped from upload.
Time (H/M in Group 1; Time column in Group 2) LENIENT If the time can't be parsed, the row still imports with 00:00. The wizard preview shows "Invalid time format" so the user knows, but Next stays enabled. Per user direction: "even not valid next button can be abled".

4.10 — tryParseTime formats supported

FormExampleResult
HH:MM (colon)"14:30"14h 30m
HH MM (space)"14 30"14h 30m
HHMM (4 digits)"1430"14h 30m
HMM (3 digits — CIMIS)"100"1h 0m
HH:MM:SS (with seconds)"14:30:45"rejected — backend doesn't carry seconds anyway
HH.MM (dot)"14.30"rejected — too easy to confuse with decimals

4.11 — The synthetic check column

Every imported row gets a column called check with default value '1'. It's added at finalize time (in handleImport), not earlier — Steps 1–3 don't see it. Per backend convention, '0' means excluded, '1' means included.

VariantStatusWhy
'1' / '0' stringChosenBackend stores values as strings; '0'/'1' are the agreed sentinel.
'true' / 'false'RejectedInitial draft used these; user corrected to 0/1.
Boolean typeRejectedValues are strings everywhere; no need for a separate type just for one column.

4.12 — Date-time column exclusion on submit

The columns the wizard ships to the backend exclude:

  • The synthetic merged "Date-Time" pseudo-column (it's not a real column in the dataset)
  • The original mapped date/time component columns (year, month, day, hour, minute for Group 1; Date, Time for Group 2)

The backend payload encodes the date/time directly in each values entry's {date, time, value}, so re-sending the source columns is redundant. date and time are also reserved column names per spec.

5. The four wizard steps

Step 1 — File Preview

Component: StepFilePreview.tsx

What's on screen: File label, read-only filename input (background #424242), Browse button.

What clicking Browse does: Dispatches importPickFileRequested(). The saga opens a native dialog (attached to the parent window so it's modal — see 4.x), reads the file, dispatches importPickFileSucceeded({ filename, rawText }). The wizard auto-parses on the next render.

Error states: Two distinct banners — "Could not open file" (file dialog/read failure) vs "Invalid file" (parse failure).

Bug fix: Cancelling the dialog used to leave Browse stuck on "Opening…". Saga now dispatches importPickFileFailed('') on cancel — empty string keeps the banner hidden but flips fileLoading back to false.

Step 2 — Data Preview

Component: StepDataPreview.tsx

What's on screen: Delimiter dropdown, Header Lines to Skip number input, Excel-style grid table (border per cell, gray #3a3a3a header background), Column Labels Preview chips (white pills, black text, 3px radius).

Reactive re-parse: Changing delimiter or skip count re-runs parseDelimited() on the cached rawText. If parsing throws, an amber banner shows the error and the previous successful parse stays visible.

XML mode: Both inputs disabled. The XML structure is parsed once at file load.

Step 3 — Date / Time

Component: StepDateTime.tsx

Mode picker: Group 1 (separate Y/M/D/H/M columns) or Group 2 (single Date column + optional Time column).

Mapping: Each role gets a column dropdown. In Group 2 mode, Date also has a date-format dropdown (DD/MM/YYYY, MM/DD/YYYY, YYYY-MM-DD, …).

Auto-detection: When the file first arrives, headers are searched for keywords (year, month, day, hour, minute, date, time) and slots are auto-filled. If Y/M/D were all found, mode defaults to Group 1.

Live preview: First 5 rows shown as Raw → Parsed (M/D/YYYY, HH:MM:SS PM/AM). Stats banner: "All N valid" / "X of N valid · Y will import as Invalid".

Gating: Next is enabled when at least one row produces a valid Date.

Step 4 — Review & Import

Component: StepReview.tsx

What's on screen: Disabled-checked Date-Time row (always required), then every non-date-time column with a checkbox (default checked). Each row shows the column name + first 3 example values.

Toggle behavior: Unchecking excludes that column from the dataset built on Import.

Click Import: Builds the ImportedDataset in handleImport(): synthetic check column at index 0 with all values '1', then the kept user columns. Records sorted ascending by dtIso (null/Invalid rows last). Dispatches importFinalizeRequested(dataset).

7. Redux + Saga structure

Actions

// File pick (saga performs file dialog + read)
IMPORT_PICK_FILE_REQUESTED   → fileLoading = true, fileError = null, pickedFile = null
IMPORT_PICK_FILE_SUCCEEDED   → fileLoading = false, pickedFile = { filename, rawText }
IMPORT_PICK_FILE_FAILED      → fileLoading = false, fileError = msg

// Finalize (saga performs the API call)
IMPORT_FINALIZE_REQUESTED    → importing = true, importError = null
IMPORT_FINALIZE_SUCCEEDED    → importing = false, dataset = payload, pickedFile cleared
IMPORT_FINALIZE_FAILED       → importing = false, importError = msg

// Reset (e.g. wizard closed)
IMPORT_RESET                 → clears all transient fields, preserves dataset

Sagas

WorkerWatcherReason
pickFileWorker takeLatest(IMPORT_PICK_FILE_REQUESTED) If user double-clicks Browse, cancel the in-flight dialog and start fresh
finalizeImportWorker takeLeading(IMPORT_FINALIZE_REQUESTED) Idempotent prevention: if a request is in flight, ignore subsequent clicks instead of cancelling — POSTs are not safe to cancel mid-flight

Selectors

All memoized via reselect:

selectFileLoading, selectFileError, selectPickedFile,
selectImporting, selectImportError, selectDataset

8. Parsing rules in depth

Delimiter detection (detectDelimiter)

The classic problem: a file with commas inside quoted fields could fool a count-based detector. Our heuristic preferences:

  1. Among the 5 candidates (,, ;, \t, |, space), find ones whose count is identical across the first 10 non-comment lines. Consistency wins over volume — a real delimiter appears the same number of times in every data row.
  2. Among consistent candidates, pick the one with the highest count.
  3. If none are perfectly consistent, fall back to the highest average count.

Header skip detection (detectHeaderLinesToSkip)

Skip while:

  • Line starts with #, //, or ;; (comment markers)
  • Line's column count differs from the modal count of the trailing 8 lines

Stop on the first line that matches the modal column count.

XML record extraction (parseXml)

  1. DOMParser the input. Reject if it produces a <parsererror> element.
  2. Find the most-repeated direct child of root — that's the record tag.
  3. Take the union of all child element names across records as the headers.
  4. For each record, output a row with values in header order; missing fields → empty string.

Strict column-count rule (parseDelimited)

If a data row has a different number of fields than the header row, we throw. Padding or truncating silently would hide data corruption. The wizard catches this and shows it on Step 1's banner.

Date strictness

Year must be exactly 4 digits in the source string ("26" is rejected even though the value 26 is technically a valid year). Removes the year-2000 ambiguity question entirely.

Test infrastructure

Three deliberate additions to src/renderer/src/tests/setup.ts:

  • ResizeObserver stub (jsdom doesn't ship one)
  • In-memory localStorage implementation (Vitest 4 web-storage shadowing fix)
  • window.api no-op stubs (so saga effect-equality assertions don't throw on property access)

What we deliberately didn't snapshot-test

The wizard's index.tsx is huge and tailwind-heavy — snapshot tests would break on every styling tweak. We test behavior (callbacks, conditional rendering, dataset shape) instead.
