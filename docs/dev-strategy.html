Section 01

Project Architecture Overview

The boilerplate follows Electron's three-process model and maps each process to a separate TypeScript compilation unit. Each process has strict boundaries — they communicate only through defined IPC channels.

Main Process
src/main/index.ts
⇄
Preload Script
src/preload/index.ts
⇄
Renderer Process
src/renderer/src/

Process Responsibilities

ProcessRuntimeResponsibilities
MainNode.jsWindow lifecycle, IPC handler registration, file system access, native dialogs, backend process management
PreloadRenderer + Node accessContext bridge — translates IPC calls into a typed window.api object; enforces the security boundary
RendererChromiumAll UI — React, Redux state, sagas, API calls, and SSE streaming

Renderer Architecture

The renderer is a React application structured around Redux with dynamic injection. The key design decisions are:

  • No React Router — navigation is driven by a Redux slice (navigationReducer). Screens are rendered conditionally in App.tsx.
  • Dynamic reducer/saga injection — each container injects its own reducer and saga on mount. This keeps the store lean and supports lazy loading.
  • Containers vs Components — containers own state and side effects; components are pure UI.
  • Saga for all async — HTTP requests and SSE streams are managed by sagas, never in components or thunks.
src/
  main/                    # Electron main process (Node.js)
  preload/                # Context bridge
  renderer/src/
    store/              # Redux store, navigation
    containers/        # Screen-level Redux containers
    components/        # Pure UI components
    utils/              # api.ts, sse.ts, inject hooks

Hot Reload Behaviour

electron-vite gives different HMR behaviour per process:

  • Renderer — full Vite HMR. Component changes reflect instantly without restarting Electron.
  • Main process — rebuilds and restarts the Electron process on save.
  • Preload — rebuilds and reloads the renderer window (triggers a soft reload, not a full Electron restart).

Section 03

TypeScript Configuration

The project uses TypeScript project references — a root tsconfig.json that delegates to two distinct configs, one per Electron process.

FileCoversKey settings
tsconfig.node.json src/main/, src/preload/ No DOM lib, @types/node, Node-compatible module resolution
tsconfig.web.json src/renderer/src/ DOM lib, jsx: react-jsx, path aliases, vitest/globals types

Path Aliases (Renderer)

These aliases are configured in both tsconfig.web.json and electron.vite.config.ts so they work at compile time and at bundle time:

AliasResolves to
utils/*src/renderer/src/utils/*
store/*src/renderer/src/store/*
containers/*src/renderer/src/containers/*
components/*src/renderer/src/components/*

Strict Mode

Both tsconfigs enable "strict": true. This means:

  • Generator functions — add : Generator return type and cast all yield results explicitly: (yield call(...)) as MyType
  • Redux actions — use UnknownAction instead of the deprecated AnyAction
  • Immer — use the named export: import { produce } from 'immer'
  • Unused variables — prefix intentionally unused parameters with _
Rule Never use moduleResolution: "bundler" — it breaks subpath exports like redux-saga/effects. Keep "moduleResolution": "node" in tsconfig.web.json.

Global Type Declarations

src/renderer/src/env.d.ts declares the Window augmentation for window.api. Any new IPC methods added to the preload must be reflected here so the renderer has full type safety when calling them.

Section 04

Main Process

The main process is the Node.js host for the Electron app. It owns all privileged operations: native OS dialogs, file system access, window management, and backend subprocess control.

File: src/main/index.ts

Window Creation

The createWindow() function creates the BrowserWindow with show: false to prevent a white flash — the window reveals itself on ready-to-show. The preload path is wired here:

webPreferences: {
  preload: join(__dirname, '../preload/index.js'),
  sandbox: false  // required for contextBridge + Node access in preload
}

Dev vs Production Loading

const isDev = !app.isPackaged

if (isDev && process.env['ELECTRON_RENDERER_URL']) {
  mainWindow.loadURL(process.env['ELECTRON_RENDERER_URL'])  // Vite dev server
} else {
  mainWindow.loadFile(join(__dirname, '../renderer/index.html'))  // built file
}

IPC Handler Registration

All ipcMain.handle() calls must be registered before app.whenReady() completes, so that the renderer can call them as soon as the window loads.

ChannelHandlerReturns
dialog:openFileOpens native file pickerstring | null (path)
dialog:saveFileOpens native save dialogstring | null (path)
fs:readFileReads file contentsstring
fs:writeFileWrites content to filevoid
backend:getStatusStub — return process state{ running, pid }
backend:startStub — spawn backend{ ok }
backend:stopStub — kill backend{ ok }

Adding a New IPC Handler

The pattern is always the same — add in main, expose in preload, call from renderer:

// 1. src/main/index.ts
ipcMain.handle('myFeature:doSomething', async (_event, arg: string) => {
  // privileged operation
  return result
})

// 2. src/preload/index.ts — add to the api object
doSomething: (arg: string): Promise<Result> =>
  ipcRenderer.invoke('myFeature:doSomething', arg),

// 3. src/renderer/src/env.d.ts — add to Window.api type
doSomething: (arg: string) => Promise<Result>

// 4. Renderer usage
const result = await window.api.doSomething('value')
Security Note Never use ipcRenderer.sendSync — it blocks the renderer. Always use ipcMain.handle / ipcRenderer.invoke (async request-response). Never expose Node.js modules directly through the context bridge.
Section 05

Preload / IPC Bridge

The preload script is the security boundary between the untrusted renderer and privileged main process code. It runs with access to both Node.js APIs and the DOM, but exposes only an explicitly typed surface.

File: src/preload/index.ts

The preload builds the api object and exposes it via contextBridge.exposeInMainWorld('api', api). After this, the renderer accesses it as window.api.

const api = {
  openFile:        (filters) => ipcRenderer.invoke('dialog:openFile', filters),
  saveFile:        (filters, defaultPath?) => ipcRenderer.invoke('dialog:saveFile', filters, defaultPath),
  readFile:        (filePath) => ipcRenderer.invoke('fs:readFile', filePath),
  writeFile:       (filePath, content) => ipcRenderer.invoke('fs:writeFile', filePath, content),
  getBackendStatus:() => ipcRenderer.invoke('backend:getStatus'),
  startBackend:    () => ipcRenderer.invoke('backend:start'),
  stopBackend:     () => ipcRenderer.invoke('backend:stop')
}

Type Safety in the Renderer

src/renderer/src/env.d.ts augments the global Window interface. Keep this file in sync with the preload — every method in api must have a matching type declaration here.

declare global {
  interface Window {
    api: {
      openFile: (filters: FileFilter[]) => Promise<string | null>
      // ... all methods typed
    }
  }
}

Context Isolation

Context isolation is enabled by default (contextIsolation: true). The preload handles the fallback for non-isolated contexts (legacy mode) with a try/catch around exposeInMainWorld. Do not disable context isolation in production.

Development Guidance

  • Only wrap ipcRenderer.invoke calls — never pass raw ipcRenderer to the renderer.
  • Validate or sanitise arguments inside the ipcMain.handle handler, not in the preload.
  • For event-based IPC (push from main to renderer), use ipcRenderer.on in preload and expose a subscription function, not the raw emitter.
  • Keep the preload thin — it is a bridge, not a business logic layer.
Section 06

Redux Store & State Management

Store Configuration — store/configureStore.ts

The store is enhanced with two custom capabilities:

  • injectedReducers — a registry of dynamically added reducers
  • injectedSagas — a registry of running saga tasks
  • runSaga — bound saga middleware runner
  • createReducer() — rebuilds the root reducer by combining static + injected slices
export interface InjectableStore extends Store<unknown, UnknownAction> {
  injectedReducers: Record<string, Reducer>
  injectedSagas:    Record<string, Task>
  runSaga:          SagaMiddleware['run']
  createReducer:    () => Reducer
}

Root Reducer — store/reducers.ts

The only statically-mounted slice is navigation. All feature slices (containers) are injected dynamically. The RootState type is derived from the reducer factory:

export type RootState = ReturnType<ReturnType<typeof createReducer>>

Because dynamic slices are not in the static type, always cast dynamically injected state as (state as any).sliceName in selectors, as shown in the container pattern.

Navigation Reducer — store/navigationReducer.ts

Replaces React Router. Add new screen names to the Screen union to extend navigation:

type Screen = 'home' | 'project' | 'settings'  // extend here

const navigationReducer = createSlice({
  name: 'navigation',
  initialState: { screen: 'home' as Screen },
  reducers: {
    navigate: (state, action: PayloadAction<Screen>) => {
      state.screen = action.payload
    }
  }
})

Dynamic Injection Hooks

Each container mounts its own slice on startup via two hooks:

// Inside container component:
useInjectReducer({ key: 'myFeature', reducer })
useInjectSaga({ key: 'myFeature', saga })

The hooks check if the slice is already injected (idempotent) so re-renders are safe. Sagas are started with takeLatest protection through the root watcher pattern, and are cancelled automatically when a new saga with the same key is injected.

Redux Version Compatibility

Old (deprecated)Current (Redux 5)
AnyActionUnknownAction
import produce from 'immer'import { produce } from 'immer'
createStoreconfigureStore (Redux Toolkit) or manual store
Pattern Selectors always use createSelector from reselect. Export one selector per state field rather than one selector for the whole slice — this ensures components only re-render when their specific field changes.
Section 07

Containers (Connected Screens)

A container is a self-contained Redux feature unit. Every container owns its own slice of the store and all the code that manages it. The file structure is fixed by convention:

containers/FeatureName/
  types.ts         # Domain types — AppStatus, StreamEvent, etc.
  constants.ts    # Action type string constants
  actions.ts      # Action interfaces + creator functions
  reducer.ts      # Immer reducer + state shape + initialState
  saga.ts         # REST workers + SSE workers + root watcher
  selectors.ts    # One memoised selector per state field
  index.tsx       # React component — useSelector + useDispatch hooks
  tests/
    actions.test.ts
    reducer.test.ts
    saga.test.ts
    selectors.test.ts
    index.test.tsx

types.ts — Domain Types

Extract shared domain types here to avoid circular imports between actions.ts and reducer.ts. Both files import from types.ts:

export interface AppStatus { version: string; uptime: number }
export interface StreamEvent { type: string; data: unknown; timestamp: number }

actions.ts — Action Creators

Define interfaces for each action then create a union type used by the reducer:

export type MyFeatureAction =
  | FetchStatusAction
  | FetchStatusSuccessAction
  | FetchStatusFailureAction
  | SseConnectAction
  | SseEventAction
  | SseDisconnectAction

reducer.ts — State & Reducer

State has two groups of fields — REST and SSE. Use Immer's produce (named export) for all mutations:

export interface FeatureState {
  // REST
  status:  AppStatus | null
  loading: boolean
  error:   string | null
  // SSE
  streaming: boolean
  streamLog: StreamEvent[]
}

saga.ts — Workers and Watcher

The saga file always contains three things: a REST worker, an SSE worker, and a root watcher:

export function* fetchStatusWorker(): Generator {
  try {
    const data = (yield call(api.get<AppStatus>, '/api/status')) as AppStatus
    yield put(actions.fetchStatusSuccess(data))
  } catch (err) {
    yield put(actions.fetchStatusFailure((err as Error).message))
  }
}

export default function* featureSaga(): Generator {
  yield takeLatest(FETCH_STATUS, fetchStatusWorker)
  yield takeLatest(SSE_CONNECT, sseWorker)
}

Export workers (not just the default watcher) so tests can step through them directly.

index.tsx — The Component

Components use hooks only — never the connect() HOC:

export function MyFeature(): React.JSX.Element {
  useInjectReducer({ key: 'myFeature', reducer })
  useInjectSaga({ key: 'myFeature', saga })

  const dispatch = useDispatch()
  const status   = useSelector(selectStatus)
  const loading  = useSelector(selectLoading)

  useEffect(() => { dispatch(actions.fetchStatus()) }, [dispatch])
  // ...
}
Section 08

API & SSE Utilities

utils/constants.ts — Base URL

export const BASE_URL =
  (window as any).__APP_BASE_URL__ ?? 'http://localhost:8000'

Override at runtime by setting window.__APP_BASE_URL__ before the app loads — useful for staging environments or when the backend port is dynamic.

utils/api.ts — HTTP Client

A thin wrapper around fetch that prepends BASE_URL, sets JSON headers, and throws a typed ApiError on non-2xx responses:

export class ApiError extends Error {
  constructor(public readonly status: number, message: string) {
    super(message)
  }
}

export const api = {
  get:    <T>(path: string) => request<T>('GET', path),
  post:   <T>(path: string, body?: unknown) => request<T>('POST', path, body),
  put:    <T>(path: string, body?: unknown) => request<T>('PUT', path, body),
  patch:  <T>(path: string, body?: unknown) => request<T>('PATCH', path, body),
  delete: <T>(path: string) => request<T>('DELETE', path)
}

Calling from a Saga

const result = (yield call(api.get<MyType>, '/api/resource')) as MyType

The generic type parameter is erased at runtime — it only provides IDE autocomplete and type checking. The saga cast as MyType tells TypeScript what to expect from the yield.

Error Handling

Catch ApiError in the saga's catch block to differentiate HTTP errors from network errors:

} catch (err) {
  if (err instanceof ApiError && err.status === 401) {
    yield put(actions.sessionExpired())
  } else {
    yield put(actions.fetchStatusFailure((err as Error).message))
  }
}

utils/sse.ts — SSE Channel Factory

Converts a browser EventSource into a Redux-Saga EventChannel. This bridges the push-based SSE model into the pull-based saga model.

export function createSseChannel<T = unknown>(
  path: string,
  namedEvents: string[] = []
): EventChannel<SseMessage<T>> {
  return eventChannel((emit) => {
    const source = new EventSource(BASE_URL + path)
    source.onmessage = (e) => emit({ type: 'message', data: JSON.parse(e.data) })
    source.onerror = () => emit(END)
    namedEvents.forEach((name) => {
      source.addEventListener(name, (e: any) =>
        emit({ type: name, data: JSON.parse(e.data) })
      )
    })
    return () => source.close()  // cleanup on channel close
  })
}

SSE Worker Pattern in Sagas

function* sseWorker(): Generator {
  const channel = (yield call(createSseChannel, '/api/events')) as EventChannel<SseMessage>
  try {
    while (true) {
      const result = (yield race({
        msg:  take(channel),
        stop: take(SSE_DISCONNECT)
      })) as { msg?: SseMessage; stop?: unknown }

      if (result.stop) break
      if (result.msg) yield put(actions.sseEvent({ ...result.msg, timestamp: Date.now() }))
    }
  } finally {
    channel.close()               // close EventSource
    yield put(actions.sseDisconnect())  // sync Redux state
  }
}

Named SSE Events

If the server sends custom event types (e.g. event: progress), pass them to the channel factory:

const channel = yield call(createSseChannel, '/api/events', ['progress', 'error'])
Section 09

Navigation System

Navigation is managed entirely through Redux — no React Router. This simplifies the architecture for a desktop app where deep linking and browser history are irrelevant.

Adding a New Screen

Step 1 — Extend the Screen type in store/navigationReducer.ts

type Screen = 'home' | 'project' | 'settings'  // add 'settings'

Step 2 — Render the screen in App.tsx

const screen = useSelector((state: RootState) => state.navigation.screen)

return (
  <div className="flex flex-col h-screen bg-dark">
    {screen === 'home'     && <HomePage />}
    {screen === 'project'  && <ProjectPage />}
    {screen === 'settings' && <SettingsPage />}
  </div>
)

Step 3 — Navigate from any component

const dispatch = useDispatch()
// ...
<button onClick={() => dispatch(navigate('settings'))}>
  Open Settings
</button>

Passing State Between Screens

Because there is no URL, inter-screen data is passed through the Redux store — not through route params. Add a context field to the navigation state if you need to pass structured data alongside navigation:

type NavigationState = {
  screen: Screen
  context: Record<string, unknown>
}
Design Note For a desktop app with a sidebar + main content layout, you may want to split navigation into two Redux slices: one for the top-level screen and one for sub-navigation within a screen (tabs, panels). Keep the top-level navigationReducer static and handle sub-navigation in the feature container's own reducer.

Styling (Tailwind CSS)

Custom Theme

The design system extends Tailwind with three custom dark-UI colours defined in tailwind.config.js:

TokenHexUsage
bg-dark#0f1117App background, code blocks
bg-panel#1a1d27Cards, panels, sidebars
border-app-border#2a2d3aAll borders and dividers
Important The token is named app-border, not border. Using border as a colour key conflicts with Tailwind's built-in border-width utilities. Always use border-app-border for border colour and bg-app-border for hover backgrounds.

Layout Conventions

  • Top-level screen wrapper: flex flex-col h-full p-6 gap-6
  • Cards / sections: bg-panel border border-app-border rounded-lg p-4
  • Primary text: text-neutral-100
  • Secondary / label text: text-neutral-400
  • Destructive / error: text-red-400
  • Monospace values: font-mono text-neutral-100

Adding Global Styles

Global styles go in src/renderer/src/index.css. The three Tailwind directives must stay at the top:

@tailwind base;
@tailwind components;
@tailwind utilities;

/* custom globals below */

Component-Level Styles

Use Tailwind utility classes directly in JSX. For conditional classes, prefer array join over template literals:

Section 11

Testing Strategy

Tests live inside the feature they test — in a tests/ subfolder within the container or component directory. This keeps tests close to the source and easy to find.

Test Runner & Setup

Vitest is configured in vitest.config.ts with jsdom environment and Testing Library setup. The globals: true setting means no explicit imports of describe, it, or expect are needed.

What to Test Per File

FileTest approachKey assertions
actions.test.ts Pure function — call creator, check output object Correct type constant, payload shape
reducer.test.ts Pure function — pass state + action, check next state Field mutations, immutability (original unchanged), initial state
saga.test.ts Generator stepping — manually call .next() / .throw() call effects match expected API call, put effects match expected actions
selectors.test.ts Pure function — build mock state, assert selector output Returns correct field, falls back to initialState when slice absent
index.test.tsx React Testing Library — render with mock store, query DOM Key elements render, snapshot regression

Saga Testing Pattern

Test saga workers by manually stepping through the generator. This avoids mocking the saga runtime and keeps tests fast:

it('dispatches fetchStatusSuccess on success', () => {
  const gen = fetchStatusWorker()

  // Step 1: should yield a call to api.get
  expect(gen.next().value).toEqual(call(api.get, '/api/status'))

  // Step 2: inject mock response, should put success action
  const mockStatus = { version: '1.0', uptime: 30 }
  expect(gen.next(mockStatus).value).toEqual(
    put(actions.fetchStatusSuccess(mockStatus))
  )

  // Step 3: done
  expect(gen.next().done).toBe(true)
})

Component Testing — Mock Store

Components require an InjectableStore-compatible mock. Build the minimum stub that satisfies the interface:

const mockStore = createStore((state = {}) => state) as InjectableStore
mockStore.injectedReducers = {}
mockStore.injectedSagas    = {}
mockStore.runSaga          = () => ({ ...} as any)
mockStore.createReducer    = () => ((s = {}) => s) as Reducer<unknown, UnknownAction>

render(
  <Provider store={mockStore}>
    <MyContainer />
  </Provider>
)

Coverage

npm run test:coverage

Coverage reports are generated by @vitest/coverage-v8 into a coverage/ directory. Focus coverage on reducers and saga workers — these carry the business logic. Component snapshot tests are supplementary.

electron-vite Build Pipeline

The build runs three Vite instances in sequence:

  1. Main — src/main/index.ts → out/main/index.js (SSR/CJS bundle, externals: electron + node builtins)
  2. Preload — src/preload/index.ts → out/preload/index.js (CJS bundle)
  3. Renderer — src/renderer/ → out/renderer/ (standard Vite client bundle with tree-shaking)

electron-builder Configuration

Packaging is configured in electron-builder.yml:

appId: com.navyug.helios
productName: HeliosApp
directories:
  buildResources: build
files:
  - out/**
  - package.json

Linux Sandbox in Packaged Builds

The --no-sandbox flag in the dev script must not be carried into production packaging. The packaged app should have the sandbox properly configured. electron-builder handles setting SUID permissions on chrome-sandbox during packaging on Linux.

Before Packaging Remove or guard the --no-sandbox flag so it only applies in dev. Consider using an environment-gated approach: check !app.isPackaged in main, or use a separate dev script only.

This section walks through the complete process of adding a new screen with a REST API integration and a real-time SSE stream.

Step 1 — Scaffold the container

npm run generate
# Choose: container → name: "ProjectDashboard" → yes to all options

The generator produces these — update the generated stubs to use your real domain types from Step 2. The REST + SSE pattern is already in place; only the type names need updating.

If the feature needs native OS access, add the IPC handler trio: main handler → preload bridge → env.d.ts type declaration.
